# ZevLink File Transfer Protocol v1

Status: implementation specification for the first ZevLink file-transfer release.

This document uses **MUST**, **MUST NOT**, **SHOULD**, **SHOULD NOT**, and **MAY**
as normative requirements.

## 1. Goals

Protocol v1 provides:

- File and directory offers between an already paired Android device and Mac.
- Explicit accept, decline, and automatic-accept decisions.
- Direct local-network transfer without a cloud relay.
- TLS-protected metadata and file contents.
- Fixed-size chunks written directly to storage with bounded memory use.
- IPv6 and IPv4 connection candidates.
- One to eight persistent data connections per transfer.
- Cancellation and resume after process, device, or network interruption.
- Per-chunk and whole-file integrity verification.

Protocol v1 does not provide compression, content deduplication, delta transfer,
or cloud fallback. Those features require separately negotiated capabilities in
a later protocol revision.

## 2. Architecture

The existing ZevLink pairing, Bonjour discovery, and presence messages are the
bootstrap and discovery plane. The existing clipboard HTTP listeners on ports
9876 and 9877 remain unchanged.

File-transfer metadata and data MUST NOT use the existing plaintext clipboard
request path. Each device instead exposes a file-transfer HTTPS service on a
dynamically selected port. The service port, protocol versions, and certificate
fingerprint hint are advertised to the paired peer through the existing
discovery plane. Discovery advertisements are not trusted enrollment records;
the fingerprint MUST be authenticated by the secure device-enrollment procedure
before it is pinned.

The device receiving files runs the HTTPS service. The sender connects to the
receiver, creates an offer, waits for a decision, and uploads accepted chunks.
The same protocol is used in both directions.

### 2.1 Transport requirements

- HTTP/1.1 over TLS 1.3 is the v1 wire transport.
- The peer certificate MUST match the fingerprint saved for the paired device.
- Requests MUST carry paired-device authentication inside TLS.
- Pairing tokens, filenames, hashes, and file bytes MUST NOT be sent over a
  plaintext connection.
- Data connections SHOULD use HTTP keep-alive.
- A transfer MAY use 1 through 8 simultaneous data connections.
- Chunk request bodies MUST be streamed. Implementations MUST NOT accumulate a
  complete file or complete transfer in memory.
- Compression MUST be disabled for chunk bodies.

The exact device-key generation and TLS enrollment procedure is defined by the
device-identity implementation step. A file-transfer implementation MUST remain
disabled until the secure transport requirements above are satisfied.

## 3. Versioning And Capabilities

Every request MUST include:

```http
ZevLink-Protocol-Version: 1
```

Every response MUST include the same header. A receiver that does not support
version 1 returns `426 Upgrade Required` with its supported versions:

```json
{
  "error": "unsupportedProtocolVersion",
  "message": "File transfer protocol version 1 is not supported.",
  "supportedVersions": [2]
}
```

Discovery advertises the following fields:

| Field | v1 value | Meaning |
| --- | --- | --- |
| `transferVersions` | `1` | Comma-separated supported major versions. |
| `transferPort` | `1...65535` | HTTPS listener port. |
| `transferCert` | lowercase hex SHA-256 | Paired TLS certificate fingerprint. |
| `transferCapabilities` | comma-separated tokens | Optional extensions. |

Unknown capability tokens MUST be ignored. The initial v1 capability tokens
are:

- `chunks-4m`: mandatory 4 MiB chunks.
- `parallel-8`: supports up to eight connections.
- `resume-ranges`: supports verified-range resume.
- `ipv6`: can publish IPv6 candidates.
- `ipv4`: can publish IPv4 candidates.

## 4. Identifiers, Limits, And Sizes

- `transferId` MUST be a lowercase UUID v4 generated by the sender.
- `fileId` MUST be a lowercase UUID v4 unique within the transfer.
- File sizes and byte offsets are unsigned 64-bit integer values encoded as JSON
  numbers. Implementations MUST reject values that cannot be represented exactly
  by their native integer type.
- The fixed v1 chunk size is 4,194,304 bytes (4 MiB).
- Every chunk except the final chunk of a file MUST be exactly 4 MiB.
- The final chunk contains the remaining bytes and MAY be empty only for a
  zero-byte file, which has no chunk upload requests.
- A transfer contains at most 10,000 entries.
- Offer JSON is limited to 4 MiB.
- A relative path is limited to 1,024 UTF-8 bytes.
- A display name is limited to 255 Unicode scalar values.
- Symlinks, sockets, device files, and other special files are not supported.

All hashes in v1 use SHA-256 and are encoded as 64 lowercase hexadecimal
characters.

## 5. Connection Candidates

The receiver returns an ordered list of candidates after accepting an offer.

```json
{
  "id": "candidate-1",
  "family": "ipv6",
  "host": "2401:db00:10::25",
  "port": 49172,
  "priority": 100,
  "networkId": "wifi"
}
```

Candidate fields:

| Field | Required | Description |
| --- | --- | --- |
| `id` | yes | Opaque identifier unique in the response. |
| `family` | yes | `ipv6` or `ipv4`. |
| `host` | yes | Address literal without brackets or DNS lookup. |
| `port` | yes | File-transfer HTTPS port. |
| `priority` | yes | Higher values are attempted first. |
| `networkId` | yes | Opaque network grouping such as `wifi` or `hotspot`. |

Receivers SHOULD prefer reachable IPv6 addresses, followed by IPv4. Loopback,
multicast, unspecified, cellular, and peer addresses outside the receiver's
active local networks MUST NOT be advertised.

Link-local IPv6 addresses require interface scoping and SHOULD NOT be advertised
in v1 unless both implementations can recover the correct scope from their
existing connection. Candidate hosts MUST never be placed directly into a URL
without structured URL construction and IPv6 bracket handling.

The sender SHOULD race the highest-priority IPv6 and IPv4 candidates with a
250 ms delay, then retain the first authenticated connection. A failed candidate
MUST NOT fail the transfer while untried candidates remain.

## 6. Transfer Manifest

An offer contains a manifest. Paths are relative to the transfer root and use
`/` as the separator on both platforms.

```json
{
  "protocolVersion": 1,
  "transferId": "3b88ee67-f1c4-46ca-a868-a494ba34571b",
  "senderDeviceId": "9721584a-8fb0-4ecc-bad5-25d2693f3d54",
  "senderName": "Android Phone",
  "createdAt": "2026-06-18T07:30:00Z",
  "nonce": "d95fe338f5a14e56837d936f4f869f23",
  "totalBytes": 1591738368,
  "entryCount": 1,
  "requestedStreamCount": 4,
  "entries": [
    {
      "fileId": "7fc9ad9e-ce8e-4cf2-b764-ce1d343c2e66",
      "kind": "file",
      "relativePath": "video.mkv",
      "size": 1591738368,
      "modifiedAt": "2026-06-17T17:01:07Z",
      "sha256": "0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef",
      "mediaType": "video/x-matroska"
    }
  ]
}
```

`kind` is either `file` or `directory`. Directory entries have no `size`,
`sha256`, or `mediaType`. Empty directories MUST be included explicitly.

`totalBytes` MUST equal the sum of all file sizes. `entryCount` MUST equal the
number of entries. The receiver MUST reject an inconsistent manifest.

Paths MUST be Unicode NFC, non-empty, and relative. A receiver MUST reject:

- Absolute paths.
- Empty, `.` or `..` components.
- NUL characters or platform path separators inside a component.
- Paths that escape the selected destination after canonicalization.
- Duplicate paths after the destination filesystem's case and normalization
  rules are applied.

## 7. HTTP API

The base path is `/zevlink/transfer/v1`.

### 7.1 Create an offer

```http
POST /zevlink/transfer/v1/offers HTTP/1.1
Content-Type: application/json
Idempotency-Key: 3b88ee67-f1c4-46ca-a868-a494ba34571b
ZevLink-Protocol-Version: 1
```

The body is the transfer manifest. Repeating the same request with the same
`Idempotency-Key` and identical body MUST return the existing transfer. Reusing
the key with a different body returns `409 Conflict`.

Possible responses:

- `201 Created`: accepted immediately by policy.
- `202 Accepted`: awaiting a user decision.
- `409 Conflict`: identifier collision or conflicting manifest.
- `413 Content Too Large`: limits or available-storage policy rejected it.
- `422 Unprocessable Content`: invalid manifest.

Accepted response:

```json
{
  "transferId": "3b88ee67-f1c4-46ca-a868-a494ba34571b",
  "state": "accepted",
  "chunkSize": 4194304,
  "streamCount": 4,
  "resumeToken": "opaque-random-value",
  "candidates": [
    {
      "id": "candidate-1",
      "family": "ipv6",
      "host": "2401:db00:10::25",
      "port": 49172,
      "priority": 100,
      "networkId": "wifi"
    }
  ]
}
```

The receiver selects `streamCount` in the inclusive range 1 through 8 and MUST
NOT exceed `requestedStreamCount`. Candidate data is populated from the
receiver's currently active local interfaces.

### 7.2 Read offer or transfer state

```http
GET /zevlink/transfer/v1/transfers/{transferId} HTTP/1.1
```

The response includes the current state, decision, candidates when accepted,
and progress totals. Senders MAY poll pending offers with exponential backoff
from 500 ms to 5 seconds.

### 7.3 Accept or decline

```http
POST /zevlink/transfer/v1/transfers/{transferId}/decision HTTP/1.1
Content-Type: application/json
```

```json
{
  "decision": "accept",
  "savePolicy": "downloads"
}
```

`decision` is `accept` or `decline`. A decline is terminal. A repeated identical
decision is idempotent. A conflicting later decision returns `409 Conflict`.

This endpoint is normally invoked by the receiver's local UI. It is part of the
protocol so the decision state has one canonical representation; remote peers
MUST NOT be allowed to approve their own offers.

### 7.4 Query verified ranges

```http
GET /zevlink/transfer/v1/transfers/{transferId}/ranges HTTP/1.1
ZevLink-Resume-Token: opaque-random-value
```

```json
{
  "chunkSize": 4194304,
  "files": [
    {
      "fileId": "7fc9ad9e-ce8e-4cf2-b764-ce1d343c2e66",
      "verifiedRanges": [
        { "startChunk": 0, "endChunkExclusive": 32 },
        { "startChunk": 40, "endChunkExclusive": 41 }
      ]
    }
  ]
}
```

Ranges are sorted, non-overlapping, and expressed in chunk indexes. The sender
MUST upload only missing chunks after a resume query.

### 7.5 Upload a chunk

```http
PUT /zevlink/transfer/v1/transfers/{transferId}/files/{fileId}/chunks/{chunkIndex} HTTP/1.1
Content-Type: application/octet-stream
Content-Length: 4194304
Content-Range: bytes 0-4194303/1591738368
ZevLink-Chunk-SHA256: 0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef
ZevLink-Resume-Token: opaque-random-value
```

The request body is exactly one chunk. The receiver MUST validate that:

- `chunkIndex`, `Content-Range`, file size, and `Content-Length` agree.
- The range starts at `chunkIndex * 4194304`.
- The body hash matches `ZevLink-Chunk-SHA256`.
- The transfer is accepted and not cancelled.
- The authenticated peer is the transfer sender.

The receiver writes to a preallocated temporary file using a positioned write.
It MUST mark the chunk verified only after the complete body is durably handed
to the destination file and its hash matches.

Responses:

- `201 Created`: newly verified chunk.
- `204 No Content`: the identical chunk was already verified.
- `409 Conflict`: a different body was supplied for an already verified chunk.
- `416 Range Not Satisfiable`: invalid or unaligned range.
- `422 Unprocessable Content`: hash mismatch.

Chunk uploads are idempotent. A sender MAY retry a chunk after a connection
failure without first querying ranges.

### 7.6 Complete the transfer

```http
POST /zevlink/transfer/v1/transfers/{transferId}/complete HTTP/1.1
Content-Type: application/json
ZevLink-Resume-Token: opaque-random-value
```

```json
{
  "manifestSha256": "abcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcdefabcd"
}
```

The manifest hash is computed over the canonical manifest representation defined
in section 8. The receiver MUST ensure every expected chunk is verified and each
whole-file SHA-256 matches the offer before atomically moving temporary files to
their final destinations.

Responses:

- `200 OK`: completed successfully.
- `409 Conflict`: chunks are missing or the transfer is not completable.
- `422 Unprocessable Content`: a whole-file or manifest hash failed.

### 7.7 Cancel the transfer

```http
POST /zevlink/transfer/v1/transfers/{transferId}/cancel HTTP/1.1
Content-Type: application/json
```

```json
{
  "reason": "cancelledBySender"
}
```

Either authenticated participant MAY cancel. Cancellation is terminal and
idempotent. The receiver MUST stop accepting new chunks and SHOULD delete partial
files promptly. A cancelled transfer cannot be resumed under the same ID.

## 8. Canonical JSON And Manifest Hash

Protocol hashes require deterministic JSON. Canonical JSON in v1 uses:

- UTF-8 encoding without a byte-order mark.
- Object keys sorted lexicographically by Unicode scalar value.
- Arrays retained in manifest order.
- No insignificant whitespace.
- Integers in shortest decimal form.
- Strings escaped according to JSON, without escaping `/`.
- Timestamps normalized to UTC with whole seconds and a trailing `Z`.

Entries MUST be sorted by `relativePath`, then `fileId`, before canonicalization.
The manifest hash excludes transport-only fields added by a receiver, including
candidate addresses, decisions, resume tokens, progress, and state.

## 9. State Machine

The canonical states are:

```text
offered -> awaitingAcceptance -> accepted -> transferring -> verifying -> completed
                  |                 |             |              |
                  +-> declined      +-------------+--------------+-> cancelled
                                        |
                                        +-> interrupted -> transferring
                                        |
                                        +-> failed
```

- `offered`: manifest was received but not yet persisted.
- `awaitingAcceptance`: receiver policy requires a user decision.
- `accepted`: destination and resources are reserved.
- `transferring`: at least one chunk request has started.
- `interrupted`: no active connections remain, but verified chunks are retained.
- `verifying`: all chunks arrived and whole-file verification is running.
- `completed`, `declined`, `cancelled`, and `failed` are terminal.

Network loss MUST transition an accepted or transferring transfer to
`interrupted`, not `cancelled`. The receiver SHOULD retain interrupted partial
data for at least 24 hours. Retention expiration transitions the transfer to
`failed` with reason `resumeExpired` and deletes partial data.

## 10. Resume Procedure

To resume:

1. Rediscover the paired receiver by device identity.
2. Establish pinned TLS to a current candidate.
3. Fetch transfer state using `transferId`.
4. Present `resumeToken` and query verified ranges.
5. Reopen up to the negotiated number of persistent connections.
6. Upload missing chunks only.
7. Call the completion endpoint again.

`resumeToken` MUST contain at least 128 bits of cryptographically random data,
MUST be bound to the transfer and paired sender, and MUST be stored as sensitive
data. It is authorization in addition to, not instead of, paired-device TLS and
request authentication.

If receiver state was lost, it returns `404 Not Found`; the sender may create a
new offer with a new transfer ID. If the destination file changed outside
ZevLink, the receiver MUST invalidate affected verified ranges before resuming.

## 11. Parallel Scheduling And Progress

The sender maintains one shared queue of missing chunk indexes. Each active
connection takes the next chunk from the queue. Implementations SHOULD begin
with one connection and increase concurrency when measured throughput improves.

Rules:

- Active stream count is always between 1 and the negotiated `streamCount`.
- Two connections MUST NOT upload the same chunk concurrently.
- Small transfers under 16 MiB SHOULD use one connection.
- Progress is the sum of verified payload bytes, not bytes read from the source
  or written to a socket.
- Retries MUST NOT double-count progress.
- Speed and ETA SHOULD use a rolling window of verified bytes.
- Receivers MAY reduce concurrency through a `429 Too Many Requests` response
  with `Retry-After` and `ZevLink-Max-Streams` headers.

The protocol permits eight streams because testing showed that high-performance
transfer engines may stripe work across multiple TCP connections. Implementations
must still benchmark 1, 2, 4, and 8 streams; more streams are not assumed faster.

## 12. Error Format

Non-success responses use JSON:

```json
{
  "error": "hashMismatch",
  "message": "Chunk SHA-256 did not match the request header.",
  "retryable": true,
  "details": {
    "fileId": "7fc9ad9e-ce8e-4cf2-b764-ce1d343c2e66",
    "chunkIndex": 12
  }
}
```

Stable v1 error codes include:

- `authenticationFailed`
- `certificateMismatch`
- `unsupportedProtocolVersion`
- `invalidManifest`
- `invalidPath`
- `insufficientStorage`
- `offerDeclined`
- `transferCancelled`
- `resumeExpired`
- `invalidRange`
- `hashMismatch`
- `missingChunks`
- `destinationChanged`
- `tooManyStreams`
- `internalError`

The `message` is diagnostic and MUST NOT be parsed. `details` MUST NOT contain
pairing tokens, resume tokens, private keys, or full local filesystem paths.

## 13. Security Requirements

- Only the paired device may create, inspect, upload, resume, complete, or cancel
  a transfer.
- Certificate pinning MUST happen before request authentication is transmitted.
- Authentication comparisons MUST be constant-time where applicable.
- Offers MUST include a cryptographically random nonce and creation time.
- A receiver MUST reject replayed offer nonces outside the idempotent request for
  their original transfer ID.
- Implementations SHOULD reject offers more than 10 minutes in the future or
  past unless clock skew is known.
- Filenames and media types are untrusted input.
- Temporary files MUST use application-private or user-approved storage and MUST
  not be executable by default.
- Final placement MUST never overwrite an existing file silently. Collision
  policy is chosen locally by the receiver.
- Logs MUST use transfer and file IDs rather than tokens or private paths.

## 14. Compatibility Rules

- Clipboard, notification, call, presence, remote-control, and AirPlay endpoints
  continue operating when file transfer is unavailable.
- A peer that does not advertise `transferVersions` is treated as not supporting
  file transfer.
- Major versions are incompatible. Minor behavior is introduced only through
  ignored capability tokens or optional JSON fields.
- Unknown JSON fields MUST be ignored unless their names begin with `required_`.
- A message containing an unknown `required_` field MUST be rejected with
  `422 Unprocessable Content`.
- Existing paired devices require a one-time secure transport enrollment before
  file transfer is enabled; clipboard pairing itself is not discarded. A
  discovery-advertised certificate fingerprint MUST NOT be trusted on its own.

## 15. Implementation Sequence

The protocol should be implemented in this order:

1. Shared models, validation, canonical JSON, and test vectors.
2. Device identities, TLS 1.3, certificate pinning, and authentication.
3. Offer, decision, cancellation, and state persistence.
4. Single-stream chunk upload and integrity verification.
5. Resume-range persistence and retry behavior.
6. Adaptive 2, 4, and 8 connection scheduling.
7. Platform share integrations and user-facing settings.
